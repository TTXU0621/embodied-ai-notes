# Easy-Vibe：Vibe Coding 实战技法提取

> 从「NanoBanana 素材生产 Agent」章节中提炼的 Vibe Coding 通用方法论。

---

## 一、增量式 Vibe Coding 四段法

该教程将完整 Agent 项目拆为 4 个独立 Prompt，逐段喂给 AI，每段只做一件事：

| 环节 | Prompt 主题 | Vibe Coding 要点 |
|------|------------|-----------------|
| **1** | Gradio Blocks 界面框架 | **先搭骨架**：只描述布局和组件，不含任何业务逻辑 |
| **2** | LLM 意图识别模块 | **接骨**：基于已有代码追加新模块，明确 API 和依赖 |
| **3** | 生图提示词生成 | **添肉**：追加业务逻辑，保持与前面模块的接口一致 |
| **4** | 图像生成执行 | **收口**：串起全部流程，前后端联动 |

> **核心原则**：永远不要让 AI 一次生成全部——把大需求拆成"界面 → 逻辑 → 集成"三段，每段独立 Prompt。

## 二、结构化 Prompt 写作模板

从教程中提取的 Prompt 五要素结构：

```
[板块名称]：清晰的功能标识

1、任务目标
· 一句话说清楚这一轮要做什么

2、技术栈要求
· 指定框架版本（如 Gradio 4.0.0+）
· 指定依赖（如 requests>=2.31.0）
· 明确禁止项（如"禁止使用 Interface 模式"）

3、核心业务规则（绝对不可偏离）
· 意图分类逻辑（三种情况的分支处理）
· 组件交互规则（哪些初始禁用、何时启用）
· 数据流向（输入→处理→输出）

4、界面/输出要求
· 精确的布局比例（左 60% / 右 40%）
· 每个组件的属性：标签、行数、占位符、初始值、交互状态
· 命名规范（组件名必须一致，后续模块依赖）

5、输出格式
· 完整可运行文件
· 代码注释清晰
· 命名与实战一致
```

> **关键**：教程中的 Prompt 无一例外都包含"绝对不可偏离"的硬性约束，这是 AI 不跑偏的锚点。

## 三、Agent 架构通用模式：思考层 + 执行层

从该教程提炼的 Agent 设计范式，**不仅适用于图像生成，可迁移到任何 Vibe Coding 项目**：

```
用户输入
    ↓
┌─────────────────┐
│  思考层 (LLM)    │  ← 分类意图、拆解任务、改写 prompt
│  决策核心         │
└────────┬────────┘
         ↓
┌─────────────────┐
│  执行层 (专项)   │  ← 只接收优化后的指令，专注执行
│  图像/代码/数据   │
└────────┬────────┘
         ↓
      输出结果
```

**适用范围**：
- 图像生成 → LLM 做 prompt 优化 + 图像模型执行
- 代码生成 → LLM 做架构设计 + 代码模型执行  
- 数据分析 → LLM 做问题拆解 + 分析引擎执行

## 四、Vibe Coding 中的状态管理

从教程中提取的 UI 状态管理技巧：

| 技巧 | 实例 | 目的 |
|------|------|------|
| **初始禁用** | 确认按钮、生成按钮初始 `interactive=False` | 防误触，强制按流程走 |
| **条件启用** | 识别意图后 → 启用确认按钮；有提示词后 → 启用生成按钮 | 引导用户走完完整流程 |
| **反馈可见** | `intent_status` 显示当前处理状态 | 让用户知道 AI 在做什么 |

> Vibe Coding 不只是"让 AI 写代码"，更是**让 AI 按你的产品逻辑设计交互流程**。

## 五、Prompt 工程核心经验

### 5.1 分层描述法

```
主体 → 材质 → 光照 → 背景 → 分辨率
```

### 5.2 关键约束词汇（图像生成 Prompt）

| 类别 | 关键词 |
|------|--------|
| 风格 | glassmorphism, clay render, flat design, Memphis art, vector illustration |
| 质感 | frosted glass, shiny plastic, glossy, rim light, soft lighting |
| 技术 | 8k resolution, macro photography, shallow depth of field, bokeh, isometric view |
| 交付 | transparent background, svg, high quality |

### 5.3 英文优先原则

图像生成模型对英文 Prompt 的理解远好于中文。在编写生图 Prompt 时，始终使用英文。

### 5.4 一致性锚定法

```
母版图 (Reference) + 变体 Prompt = 风格一致的素材矩阵
```

只改主体描述，不改风格描述，让模型在材质、配色、光影上自动继承。

## 六、可迁移的 Vibe Coding 心法

| # | 心法 | 来源 |
|---|------|------|
| 1 | **先跑起来，再理解** | 第 1 章：不追求理解代码，只要能运行 |
| 2 | **原生 → Agent → 自建** | 三步进化：手工调 API → 用现有 Agent 工具 → 自己造 Agent |
| 3 | **拆解是 Vibe Coding 的核心技能** | 把完整 Agent 拆成 4 个独立 Prompt |
| 4 | **硬约束比软建议有效** | Prompt 中"绝对不可偏离"比"建议"更能约束 AI |
| 5 | **命名一致性是模块间契约** | 组件名在多个 Prompt 中保持一致，否则 AI 会断开 |
| 6 | **思考层可复用** | LLM 意图分类 + Prompt 生成 = 通用"智能路由器"模式 |
